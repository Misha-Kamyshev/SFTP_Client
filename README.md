# SFTP Sync Client

GUI-приложение на `Python 3.11+` и `PySide6`, которое подключается к SFTP-серверу, отслеживает изменения в локальной директории и загружает новые или изменённые файлы в выбранную директорию на сервере.

## Возможности

- экран авторизации по паролю или SSH-ключу;
- поддержка настраиваемого SFTP-порта с дефолтом `22`;
- отдельный экран управления синхронизацией;
- выбор удалённой директории через браузинг SFTP;
- сворачивание окна в системный трей вместо завершения приложения;
- автоматическое определение светлой/тёмной темы Linux;
- динамическое переключение темы без перезапуска приложения;
- запуск и остановка фоновой синхронизации без блокировки GUI;
- журнал событий и индикаторы статуса подключения/синхронизации;
- восстановление настроек и состояния после перезапуска приложения;
- автозапуск для Linux и Windows;
- безопасное хранение пароля/пути к ключу через `keyring`, с platform-specific fallback.

## Установка

```bash
python3.11 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Для корректной работы системного трея и `keyring` на Linux обычно нужны:

- графическая сессия с поддержкой `QSystemTrayIcon`;
- установленный Secret Service backend, например `gnome-keyring` или `kwallet`.

Для Windows дополнительный backend `keyring` зависит от окружения пользователя. Если он недоступен, приложение использует локальный fallback-файл в пользовательском профиле.

## Темы и внешний вид

- темы реализованы централизованно в `app/services/theme_manager.py`;
- поддерживаются `light` и `dark`;
- приложение пытается определить тему через `GTK_THEME` и `gsettings`;
- если определить тему не удалось, используется светлая;
- изменения темы отслеживаются в рантайме, приложение переключает палитру и stylesheet без перезапуска;
- цвета не зашиты в отдельных виджетах, а находятся в одном словаре внутри `ThemeManager`.

## Как работает синхронизация

1. После подключения выбираются локальная и удалённая директории.
2. При запуске выполняется первичная односторонняя синхронизация `local -> remote`.
3. Затем `watchdog` отслеживает изменения в локальной директории.
4. Для уже загруженных файлов хранится снимок `mtime + size`, чтобы не отправлять неизменённые файлы повторно.
5. При ошибке загрузки приложение пытается переподключиться и повторить отправку.

Поле порта на экране авторизации можно оставить пустым. В этом случае приложение автоматически использует `22`.

## Где хранятся настройки

- обычные настройки интерфейса и путей: `QSettings` в пользовательском профиле Qt;
- автозапуск:
  - Linux: `~/.config/autostart/sftpsyncclient.desktop`
  - Windows: `%APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup\SFTPSyncClient.cmd`
- fallback credentials file:
  - Linux: `~/.config/SFTPSyncClient/credentials.ini`
  - Windows: `%APPDATA%\SFTPSyncClient\credentials.ini`

Расположение `QSettings` зависит от окружения Qt, но обычно это что-то вроде:

```text
~/.config/LocalTools/SFTPSyncClient.conf
```

## Как хранятся пароль и ключ

- основной режим: через `keyring`, то есть системное безопасное хранилище платформы;
- fallback: если `keyring` недоступен, приложение сохраняет чувствительные данные в отдельный `INI`-файл с правами `0600`.

Это менее безопасно, чем Secret Service, но соответствует требованию восстановления состояния между перезапусками и при этом не хранит пароль в обычном JSON-файле.

Поддерживаемые типы SSH private key:

- `Ed25519`
- `ECDSA`
- `RSA`

`DSA/DSS` ключи не поддерживаются. Для новой настройки рекомендуется использовать `Ed25519`, либо `RSA`, если это требуется инфраструктурой.

Если приватный ключ защищён `passphrase`, текущий MVP показывает понятную ошибку, но отдельный ввод `passphrase` пока не реализован.

## Как работает автозапуск

После первой успешной настройки автозапуск включается по умолчанию.

На Linux приложение создаёт `.desktop`-файл в:

```text
~/.config/autostart/sftpsyncclient.desktop
```

В `Exec=` записывается актуальная команда запуска приложения. После входа пользователя в Linux-сессию окружение рабочего стола запускает этот `.desktop`-файл автоматически.

На Windows приложение создаёт `.cmd`-файл в Startup-папке текущего пользователя и запускает текущее приложение через `start`.

Автозапуск можно отключить чекбоксом на странице синхронизации.

## Сборка

### Windows `.exe`

Сборка выполняется на Windows:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\build_exe.ps1
```

Результат будет в `dist/SFTPSyncClient/` или `dist/SFTPSyncClient.exe` в зависимости от режима PyInstaller.

### Linux `.deb`

Linux-пакет собирается как standalone bundle через `PyInstaller`, а затем этот bundle упаковывается в полноценный `.deb`.
Перед сборкой нужен Python c build-зависимостями:

```bash
python3 -m pip install '.[build]'
python3 -m pip install -r requirements.txt
```

Сборка `.deb`:

```bash
chmod +x ./build.sh
./build.sh
```

Что делает сборка `.deb`:

- собирает Linux standalone bundle через `PyInstaller` из `main.py` с флагом `--windowed`;
- создаёт директорию `pkg/` со структурой Debian-пакета;
- кладёт готовый исполняемый файл и bundled runtime в `/opt/sftp-sync-client/`;
- создаёт симлинк `/usr/bin/sftp-sync-client -> /opt/sftp-sync-client/sftp-sync-client`;
- вычисляет системные runtime-зависимости бинарника через `dpkg-shlibdeps`, чтобы `apt install ./package.deb` подтягивал нужные системные библиотеки автоматически;
- создаёт desktop entry, который запускает `/opt/sftp-sync-client/sftp-sync-client`, а не `main.py` через системный `python`;
- собирает пакет командой `dpkg-deb --build`;
- прогоняет проверки через `file`, `dpkg-deb --contents` и smoke-test бинарника.

Что включено в пакет:

- PyInstaller bootloader;
- встроенный Python runtime;
- Python-зависимости приложения, включая `PySide6`, `paramiko`, `watchdog`, `keyring`;
- код приложения и сопутствующие runtime-ресурсы.

Почему после установки не нужен `pip install`:

- `.deb` больше не рассчитывает на системный Python и не запускает исходники напрямую;
- пользователь запускает bundled executable, внутри которого уже упакованы Python runtime и Python-зависимости;
- на целевой Linux-машине Python не требуется: для Python-runtime пакет самодостаточен, а системные shared libraries ставятся обычным механизмом зависимостей `apt`;
- за счёт этого после установки и после перезагрузки приложение должно открываться из меню приложений без `ModuleNotFoundError` на `PySide6` и другие pip-зависимости.

## Ограничения текущего MVP

- пока реализована только односторонняя синхронизация `локальная директория -> SFTP`;
- удаления локальных файлов не удаляют файлы на сервере;
- конфликтов версий и двусторонней синхронизации пока нет;
- сравнение файлов делается по локальному снимку `mtime + size`, без контрольных сумм;
- браузер удалённых директорий показывает только папки;
- полноценная поддержка упаковки требует сборку `.exe` на Windows и `.deb` на Linux;
- определение системной темы сейчас полноценно реализовано только для Linux.
